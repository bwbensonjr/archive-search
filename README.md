# Internet Archive Search — Aadam Jacobs Collection

A catalog and browsable website for the
[Aadam Jacobs collection](https://archive.org/details/@aadam_jacobs_collection)
of live music recordings on the Internet Archive.

The collection holds ~2,900 recorded shows spanning 1984–2023. This project
fetches their metadata, cross-references each band and venue with Wikipedia,
and publishes a static site (GitHub Pages) for browsing by band and by year.

## Data pipeline

The site is generated by three scripts that each read/write `shows.db`
(a local SQLite database):

| Step | Script | What it does |
|------|--------|--------------|
| 1. Fetch | `uv run fetch_shows.py` | Searches the IA collection with the [`ia`](https://archive.org/developers/internetarchive/cli.html) CLI and (re)builds the `show` table. |
| 2. Enrich | `uv run enrich.py` | Looks up each band/venue on Wikipedia + Wikidata and caches results in `band_info` / `venue_info`. |
| 3. Build | `uv run build_site.py` | Joins everything and writes `docs/data/*.json` for the website. |

To refresh the published site, run the three steps in order and commit the
changes under `docs/`.

### Automated daily updates

`.github/workflows/update-collection.yml` runs the full pipeline (fetch →
enrich → build) once a day, and commits the regenerated `docs/data/*.json` and
`shows.db` back to `main` only when the collection has actually changed. The
collection is in an active upload phase (new shows every few days), so a daily
cadence keeps the site close to current; change the `cron` line to adjust.
You can also trigger it manually from the repo's **Actions** tab
(*Update collection → Run workflow*). Public-collection search needs no
Internet Archive credentials, so the workflow requires no secrets.

### Enrichment details

`enrich.py` is incremental — it skips names already cached, so re-runs are
cheap and you can interrupt/resume it. Useful flags:

```bash
uv run enrich.py --bands --limit 20   # top 20 bands only (quick test)
uv run enrich.py --venues             # venues only
uv run enrich.py --refresh            # ignore cache, re-fetch everything
```

Band/venue matching is name-based and therefore fuzzy. A music-relevance
heuristic rejects implausible matches (marking them "no match" rather than
storing a wrong article). To correct a specific match, edit **`overrides.json`**:

```json
{
  "Some Band": "Exact_Wikipedia_Article_Title",
  "Mismatched Venue": ""
}
```

A real title forces that article; an empty string `""` suppresses a bad
auto-match. Names listed in `overrides.json` are always re-fetched.

## The website (`docs/`)

A dependency-free, client-side single-page app — plain HTML/CSS/JS that loads
the generated JSON and renders everything in the browser (hash-based routing,
so deep links survive a refresh):

- **Bands** (`#/`, the landing page) — collection stats plus a searchable,
  sortable list; each band page shows its Wikipedia blurb/thumbnail and a
  chronological list of its shows.
- **New** (`#/new`) — the most recently added shows, grouped into
  "Added \<date\>" sections by Internet Archive publish date.
- **Timeline** (`#/timeline`) — shows per year + each year's top bands; click a
  year to drill in.
- **About** (`#/about`) — goals and source attribution.
- **Venues** (`#/venue/<slug>`) — reachable from show rows, with Wikipedia info.

### Preview locally

```bash
python -m http.server -d docs 8000
# open http://localhost:8000
```

### Publish on GitHub Pages

Repo **Settings → Pages → Build and deployment → Deploy from a branch**, then
select branch `main` and folder **`/docs`**. No build action is required — the
files in `docs/` are served as-is.
